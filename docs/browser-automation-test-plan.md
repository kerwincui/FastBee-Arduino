# FastBee-Arduino 浏览器自动化测试方案（完整版）

## 1. 测试概述

### 1.1 测试目标
对 FastBee-Arduino 设备端 Web 管理系统进行端到端功能验证，模拟真实用户操作全流程，覆盖所有页面的增删改查操作、功能修改后的正确应用、异常场景处理以及跨模块联动验证。

### 1.2 测试框架与工具链
| 工具 | 用途 |
|------|------|
| **Playwright** (主框架) | 浏览器自动化，多设备串行测试 |
| **串口监听脚本** | 设备重启/日志验证辅助 |
| **MQTT.fx / MQTTX** | MQTT 消息收发验证 |
| **截图对比工具** | UI 回归测试 |

### 1.3 测试设备环境

| 端口 | 设备型号 | 芯片 | Flash / PSRAM | 角色 |
|------|----------|------|---------------|------|
| COM15 | ESP32-S3-F16R8 | ESP32-S3 | 16MB / 8MB PSRAM | 主测设备（全功能） |
| COM13 | ESP32-C3-F4R0 | ESP32-C3 | 4MB / 0 | 精简设备（无PSRAM） |
| COM11 | ESP32-C6-F4R0 | ESP32-C6 | 4MB / 0 | 精简设备 |
| COM16 | ESP32-F4R0 | ESP32 | 4MB / 0 | 经典设备 |

### 1.4 测试前置流程（每轮测试前执行）

```
Step 1: 擦除固件    → esptool.py --port COMx erase_flash
Step 2: 烧录固件    → platformio run -e <env> --target upload
Step 3: 上传文件系统 → platformio run -e <env> --target uploadfs
Step 4: 串口监听    → 等待 "WebServer started" 日志输出
Step 5: 获取设备IP  → 从串口日志提取 STA IP 或 mDNS 地址
Step 6: HTTP健康检查 → GET /api/health 返回 200
```

### 1.6 通用断言规则
- 每次页面导航后自动截图存档（`screenshots/<suite>/<case>.png`）
- 保存操作后断言成功消息出现（`.message-success` visible）或 API 返回 200
- 删除操作后断言列表条目消失 + 确认弹窗正确关闭
- 表单提交后等待设备响应（嵌入式设备延迟，默认 `waitForTimeout(2000)`）
- 所有等待超时默认 30s，MQTT 连接等待 60s，设备重启等待 120s
- 模态框打开后验证标题正确、表单字段预填充正确
- 模态框关闭后验证遮罩层消失（`.modal` 无 `display: block`）

## 3. 执行策略

### 3.1 执行优先级

| 优先级 | 套件 | 说明 |
|--------|------|------|
| P0-冒烟 | Suite-01, Suite-02, INT-020 | 每次构建必跑，确保基础功能可用 |
| P1-核心 | Suite-03, Suite-04, Suite-07, Suite-16 | WiFi/设备/MQTT/联网切换 是设备核心功能 |
| P2-功能 | Suite-05, Suite-06, Suite-08, Suite-10, Suite-17 | 外设/执行/Modbus/规则/性能 是开发环境功能 |
| P3-辅助 | Suite-09, Suite-11, Suite-12, Suite-13, Suite-18 | 大屏/日志/文件/用户/联动 |
| P4-回归 | Suite-14, Suite-15 | 集成验证和UI回归 |

### 3.2 多设备串行策略

```
设备循环:
  FOR each device IN [ESP32-S3, ESP32-C3, ESP32-C6, ESP32]:
    1. 烧录固件 + 上传文件系统
    2. 运行 P0 冒烟测试
    3. IF pass → 运行 P1 核心测试
    4. IF pass → 运行 P2 功能测试 (仅 ESP32-S3 运行全部)
    5. 运行 P3 辅助测试
    6. 收集测试报告
```

### 3.3 失败恢复策略

```
测试失败时:
  1. 自动截图存档 → screenshots/failures/<case-id>.png
  2. 保存页面 HTML → artifacts/failures/<case-id>.html
  3. 记录控制台日志 → artifacts/failures/<case-id>-console.log
  4. 尝试恢复:
     a. 页面级: 刷新页面，重试一次
     b. 设备级: GET /api/health 检查设备可达性
     c. 固件级: 重启设备，等待恢复，重跑当前case
  5. 如仍失败 → 标记 SKIP，继续下一用例
```
